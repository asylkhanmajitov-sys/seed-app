# Архитектура — верхний уровень

*Документ для разговора с архитектурным консультантом и для самого процесса разработки. Описывает, что и почему мы выбираем — не пошаговую инструкцию.*

---

## Главный принцип

**Максимум managed-сервисов, минимум собственной инфраструктуры.**

Один человек с Claude Code может построить много, но **не может одновременно поддерживать** Postgres-кластер, Redis, поиск, WebSocket-сервер, email-рассыльщик и платёжную систему. Поэтому всё, что можно купить как сервис — покупаем.

Это не лень. Это **правильный архитектурный выбор для соло-разработчика в 2026**. И его поддерживают сами Anthropic, Vercel, Supabase — все они построены на managed-сервисах поверх managed-сервисов.

## Слой 1 · Фронтенд

### iOS — нативно

**SwiftUI + UIKit где нужно.** Не React Native, не Flutter.

Причина: манифест говорит *«качество ощущается»*. Анимации, жесты, типографика, скорость скролла на iOS — это разница между «нормальное приложение» и «приложение, которое люди показывают друзьям». Native — единственный путь туда.

Claude Code пишет Swift на хорошем уровне. Оставшийся gap — точечные консультации с iOS-разработчиком (10–20 часов за весь Фаза 1).

### Android

**Kotlin + Jetpack Compose.** Запускается во **второй половине Фазы 2**, не сразу.

### Web

**Next.js (App Router) + React + Tailwind.** Параллельно с iOS — потому что страница профиля и подборки **должны быть в вебе с первого дня**, чтобы их можно было шерить ссылкой.

Веб не заменяет приложение — он работает как «публичная витрина» и легковесный режим чтения.

### TV (Фаза 3)

tvOS (Apple TV), Android TV. Отдельные простые приложения — не основная инвестиция.

## Слой 2 · Бэкенд и данные

### Платформа: Supabase

Один сервис закрывает: Postgres, Auth, Realtime (WebSocket), Storage (S3-совместимое), Edge Functions, pgvector для AI-эмбеддингов.

**Альтернативы рассматривались:** Firebase (привязка к Google, дороже на масштабе), AWS Amplify (сложнее), своя инфраструктура (нет смысла на старте).

**Когда мигрировать:** при ~100k активных пользователей, если упрёшься в лимиты. До этого — Supabase покроет всё.

### База данных: PostgreSQL

Внутри Supabase. Один источник правды для всех данных. Главные таблицы:

- `users`, `profiles`
- `films`, `books`, `actors`, `directors` (метаданные с TMDB/Open Library, кэшированные)
- `lists` (подборки), `list_items`
- `ratings`, `reviews`, `quotes`
- `follows`, `messages`, `notifications`
- `taste_vectors` (через pgvector, для AI-рекомендаций)

### Поиск

**Meilisearch Cloud** или **Algolia**.

Meilisearch дешевле и open-source, Algolia зрелее и дороже. На старте — Meilisearch, мигрировать в Algolia при необходимости легко.

### Кэш

**Upstash Redis.** Serverless, платишь за запросы. Для горячих рекомендаций, сессий, rate limiting.

## Слой 3 · AI

### LLM: Claude API (Anthropic)

Используем **Sonnet 4.6 для большинства задач**, **Opus 4.7 для сложных**:

- Sonnet: повседневные рекомендации, диалог, follow-up вопросы
- Opus: сложные кросс-медийные рекомендации, AI-оформление профиля, медиатор для пар

Function calling для подтягивания данных из Postgres в реальном времени (вкусы пользователя, что смотрели друзья, доступность фильмов).

### Эмбеддинги: Voyage AI

Партнёр Anthropic, специализируется на качественных эмбеддингах. Альтернатива — OpenAI text-embedding-3.

Используются для:
- Taste vectors (профиль вкуса каждого пользователя)
- Семантический поиск *«фильмы как X»*
- Соседи по вкусу (cosine similarity между векторами)

Хранение — в pgvector внутри Postgres. Не требует отдельной vector-базы вроде Pinecone.

### Кэширование AI-ответов

Критично для экономики. Если 1000 пользователей просят *«какой-нибудь нуар»* — это **не должны быть 1000 запросов к Claude**. Хешируем нормализованный промпт + контекст и кэшируем результат на 24 часа в Redis.

Без кэширования — продукт убыточен. С кэшированием — цена снижается в 10–20 раз.

## Слой 5 · Аутентификация

**Clerk** или **Supabase Auth.**

Clerk зрелее и удобнее (JWT, magic links, OAuth, MFA из коробки), Supabase Auth дешевле и встроен. Стартую с Supabase Auth, мигрирую в Clerk если нужны продвинутые фичи (например, organization management для семейных подписок в Фазе 3).

Обязательные методы входа:
- **Apple Sign In** (требуется Apple, без него App Store отклонит)
- **Google Sign In**
- **Magic link** на email
- **Phone OTP** (важно для рынков СНГ, где SMS воспринимается естественнее email)

## Слой 6 · Платежи

Двойная система — потому что Apple/Google требуют использовать **их** биллинг для подписок внутри приложения, но веб-сайт может принимать прямые карты.

- **iOS / Android in-app:** StoreKit (iOS), Google Billing
- **Web:** Stripe (международные карты)
- **СНГ-карты:** CloudPayments или ЮKassa (для тех, у кого Stripe не работает)

Сервис-абстракция (RevenueCat) сильно упрощает синхронизацию подписок между платформами. Платный, но окупается.

## Слой 7 · Real-time и уведомления

### Чат и live-обновления

**Supabase Realtime** (PostgreSQL replication через WebSocket). Хватит до десятков тысяч одновременных подключений. Дальше — Pusher или собственный WebSocket-сервер.

### Push-уведомления

**OneSignal** — кросс-платформенный (iOS APNS + Android FCM + Web Push в одном). Дешевле, чем держать APNS/FCM напрямую.

Уведомления делятся на:
- **Транзакционные** (новое сообщение, ответ на комментарий) — мгновенные
- **Digest** (weekly summary действий друзей) — агрегированные
- **Smart triggers** (post-watch follow-up, напоминание про сохранённое) — отложенные через очередь задач

### Очередь задач

**Inngest** или **Trigger.dev.** Serverless workflow-движки. Не хочется поднимать свой Redis Queue / Celery / BullMQ.

Используются для:
- Отправки post-watch пушей через 2.5 часа
- Weekly digest по пятницам
- Отложенных напоминаний *«не забудь посмотреть»*
- AI-генерации профильных тем (могут быть медленными)

## Слой 8 · Хостинг и деплой

### Веб

**Vercel.** Next.js хостится буквально в один клик, автоматический CI/CD из GitHub, edge-функции близко к пользователям.

### Бэкенд

Edge Functions внутри Supabase + Vercel API routes для серверной логики. Оба serverless — не платишь за idle.

### Файлы и медиа

**Cloudflare R2** для пользовательского контента (аватары, фото цитат). Дешевле S3, без egress-комиссий.

CDN — Cloudflare поверх всего.

### iOS / Android

Сборки и деплой через **Fastlane** + GitHub Actions. Один скрипт = новая версия в TestFlight / Internal Testing.

## Слой 9 · Безопасность

Базовый уровень закрывают managed-сервисы. Дополнительно:

### Защита от атак

- **Cloudflare WAF** (Web Application Firewall) — DDoS, базовые атаки
- **Rate limiting** через Upstash на критичных эндпоинтах (логин, AI-вызовы, регистрация)
- **CSP headers**, HTTPS везде, secure cookies

### Управление секретами

- **Doppler** или встроенный Vercel Environment Variables
- Никогда — секреты в коде или в репозитории

### Шифрование

- На уровне БД — Supabase шифрует at-rest по умолчанию
- В transit — TLS 1.3 везде
- Чувствительные поля (если будут) — отдельное приложенческое шифрование

### Постоянное сканирование

- **Claude Code Security** — встроен в workflow, сканирует код на каждом PR
- **Snyk** или **Dependabot** — для уязвимостей в зависимостях
- **Sentry** — отслеживает реальные ошибки и аномалии

### Перед публичным запуском

- Профессиональный pen test ($5–15k) — обязательно
- Compliance review для целевых стран ($1–3k)

## Вопросы для архитектурного консультанта

Эти вопросы стоит обсудить с приглашённым senior-разработчиком на этапе ревью **до начала кодинга**:

1. **Supabase vs альтернативы.** Учитывая ожидаемую нагрузку, не упрёмся ли мы в лимиты? Когда планировать миграцию?

2. **Real-time на масштабе.** Supabase Realtime хватит для 50k одновременных пользователей в чате, или сразу проектировать с Pusher / своим WebSocket?

3. **Структура БД для подборок и кросс-медиа.** Как лучше — общая таблица `items` с типом или отдельные `films` / `books`? Trade-off гибкости vs query-производительность.

4. **Стратегия кэширования AI.** Какая нормализация промптов даст максимум cache hit rate без потери релевантности?

5. **Локализация и compliance.** В каких юрисдикциях обязательна data residency (хранение данных в стране пользователя)? Это меняет всю архитектуру.

6. **Multi-tenant подписки** для пар / семей в Фазе 3 — как заложить структуру сейчас, чтобы не переделывать.

7. **Стратегия на iOS-первый.** Не будет ли проблем с фичами, которые сложно реализовать кросс-платформенно (например, AI-чат с потоковым ответом)?

8. **План миграции при переходе из Фазы 2 в Фазу 3** — где ожидаются переписывания, что закладываем сейчас.

## Что осознанно НЕ делаем

- **Не пишем свой бэкенд с нуля на Node/Go/Python без managed-БД.** Соло-разработчик не должен поддерживать Postgres-кластер.
- **Не используем Firebase.** Привязка к Google, дорогие исходящие данные.
- **Не делаем микросервисы на старте.** Монолит на Supabase Edge Functions — правильный выбор для Фаз 1–2.
- **Не пишем свой ML.** AI-слой целиком — на Claude API.
- **Не делаем кросс-платформенный фреймворк (React Native / Flutter).** Качество не то.

## Принципиальное решение, которое нужно принять перед стартом

**Выбор № 1:** монолит на Supabase + Vercel **(рекомендуется)** — быстрый старт, низкие затраты, лёгкая поддержка соло-разработчиком.

**Выбор № 2:** AWS / Google Cloud с микросервисами **(не рекомендуется)** — гибче на масштабе, но нереалистично для соло+AI.

После роста до 100k+ пользователей при необходимости — мигрируем компонентами.
